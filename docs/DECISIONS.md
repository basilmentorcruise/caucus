# Architecture Decision Records

Each ADR captures a decision and the reasoning. Decisions are **accepted** unless marked otherwise. Superseding an ADR means adding a new one that references it — don't silently rewrite history.

> **History note.** Caucus began as a project codenamed *Agora* — an "IRC-like commons for agents." Validation found that concept was already shipped by [scuttlebot.dev](https://scuttlebot.dev) (single-operator fleet) and that the name "Agora" was unusable (owned by Agora.io; an existing agent-comms protocol paper). The project pivoted to a **multi-principal war room for investigations** and was renamed **Caucus**. ADRs below are the *current* Caucus decisions; the earlier Agora ADRs are superseded.

Status: ✅ Accepted · 🕓 Deferred (planned) · 🔄 Superseded

---

## ADR-C1 — Multi-principal war room, not a single-operator fleet ✅
**Decision.** Caucus is built around multiple humans, each owning and steering their own Claude Code agent, collaborating in one shared channel. Positioning wedge: **investigations & escalations**, flagship scenario **production-incident response**.
**Alternatives.** Single-operator fleet console (scuttlebot's space); generic "agent collaboration" with no wedge.
**Why.** The single-operator commons is taken and undifferentiated. The unclaimed, valuable space is *multiple principals coordinating* — which is exactly the incident scenario, where redundant work and trapped context are most expensive and most visible. The wedge sharpens the design (ephemeral channels, claim-dedup, "humans are the real-time layer") rather than constraining it.
**Amendment (2026-06-03).** (a) **Beachhead:** production-incident response is the *headline vision and demo*, but first real usage is proven on **lower-tempo investigations** (hard debugging, security investigation, migration) to avoid the max-friction-at-worst-moment adoption tax of live incidents. (b) **Differentiation re-anchored vs [airc](https://github.com/CambrianTech/airc)** (closest prior art — multi-user "IRC for AI agents," with a claims-as-leases work-coordination protocol on its `canary` branch). Our coordination edge is narrowing, so the defensible distinction is: **(1) multi-principal agent→human identity binding** (airc identity is per-directory, TOFU, *no human binding* — genuine whitespace; our real moat); **(2) investigation/escalation domain + a typed finding/claim/question schema** (airc's coordination is dev-PR kanban); **(3) MCP-native** (server tools + turn-start hook) vs their CLI + monitor-stream. Multi-principal identity is the *wedge framing, not an absolute moat* — defensibility target is the structured investigation/postmortem record. See [CAU-21](https://github.com/basilmentorcruise/caucus/issues/21).

## ADR-C2 — Substrate: lightweight purpose-built backbone, not an Ergo fork ✅ / 🕓(spike)
**Decision.** Build a small purpose-built backbone (append-only log, claim ledger, cursors, seatbelts) behind an implementation-agnostic interface — **pending the M0 spike** (see Roadmap) confirming no core property is surprisingly hard. Keep an Ergo-backed adapter as a reversible fallback.
**Alternatives.** Fork Ergo (the original Agora plan).
**Why.** Ergo's payoff was "agents are IRC clients → channels/history/clients for free." Caucus's clients are Claude Code sessions over MCP, not IRC clients, so that payoff evaporates; what's left (free IRC-client observability) is a nice-to-have. An Ergo fork adds a protocol hop, a Go fork for claim+dedup, and a 4094-byte tag budget — cost without benefit here. A purpose-built service maps 1:1 to our primitives and is less total code for *this* concept. Reversible via the shared interface. *Supersedes the Agora "fork Ergo" decision.*

## ADR-C3 — Integration: MCP server + Claude Code hook ✅
**Decision.** Agents reach Caucus through an MCP server (post/read/claim/subscribe/discover). A Claude Code turn-start hook auto-injects new-since-checkpoint messages for passive awareness.
**Alternatives.** Native SDK only (more adoption friction); rely on the agent remembering to call a read tool (unreliable).
**Why.** Claude Code is already an MCP client, so "point your session at Caucus" is near-zero friction. The hook solves the core problem that turn-based agents won't reliably poll — awareness becomes automatic without a persistent listener. Native SDK is deferred (M2).

## ADR-C4 — Turn-based + checkpoint reads; humans are the real-time layer ✅
**Decision.** No sub-second autonomous bus. Agents catch up at turn start (via the hook) and before claiming work. The humans — present and fast during an incident — supply real-time reaction.
**Alternatives.** Background listener interrupting agents mid-task for true real-time.
**Why.** Claude Code is turn-based and human-driven; a mid-task interrupt is the jankiest, hardest part and can derail an agent. The incident scenario *has* humans in the loop, so we don't need agents to be real-time. Mid-task reactivity is a deferred enhancement for unattended cases.

## ADR-C5 — Claim-before-you-work as the dedup primitive ✅
**Decision.** Agents `claim(target)` before starting a sub-task. The claim ledger is **first-write-wins and atomic**; a granted claim is also posted as a `claim` message so the hook surfaces it. Claims are advisory-but-visible (the server records truth; agents are nudged to respect it), not hard locks.
**Alternatives.** No coordination (collisions); hard locks (brittle, deadlock-prone).
**Why.** "Stop debugging the same thing twice" is the headline value; it needs a concrete mechanic. First-write-wins is trivially correct with a single-writer backbone. Soft-but-visible claims fit a fast-moving investigation better than locks.
**Amendment (2026-06-03).** Adopt a **lease-with-TTL** claim model in the schema (claim carries an optional TTL + heartbeat; a lapsed lease frees the target) — borrowed from airc's `canary` coordination protocol, which avoids stuck claims when an agent dies mid-task. **MVP scope is still first-write-wins only** (CAU-7); TTL/heartbeat *enforcement* and release/reassignment are the claim lifecycle in CAU-18 (M2). The schema fields ship now (see MESSAGE_SCHEMA) so we don't re-version later.

## ADR-C6 — Posting verbosity is configurable per channel; default quiet 🔄 (supersedes "autonomous by default")
**Decision.** Agents post typed findings/claims/questions, but **posting verbosity is a per-channel setting — `quiet` / `normal` / `chatty` — defaulting to `quiet`** (post only consequential findings/claims/blockers; bias toward silence). The human can always tell their agent to share more, or raise the channel's verbosity.
**Alternatives.** Autonomous-by-default for every typed event (original decision — too noisy); human-must-trigger every share (people forget under pressure).
**Why.** Channel noise is the sticky trust-killer: once humans learn to ignore the feed, the whole passive-awareness UX collapses, and that habit doesn't come back. Defaulting quiet protects the calm-signal promise; making it configurable lets a team dial up verbosity for a hot incident without hard-coding the trade-off. Post-volume is an anti-metric, not a goal. *Supersedes the original "share autonomously by default."*
**Addendum (CAU-94, 2026-06-10).** The hook's **per-message render budget** is likewise a per-channel knob — `renderBudgetChars`, default `200` (byte-identical to the prior hard-coded budget, so the calm feed is unchanged). A channel may raise it (up to `INJECTED_DELTA_CAP_CHARS`) for evidence-heavy work; a truncated body now carries an explicit `+truncated, N chars — caucus_read_channel` affordance rather than dropping its tail silently. This is an addendum, not a reversal: the default stays conservative and the overall delta cap is unchanged.
**Addendum (CAU-92, 2026-06-10).** Cross-room posting is **permitted but join-gated and quiet-by-default**: a session may post (or claim) into a room other than its `CAUCUS_CHANNEL` home, but ONLY one it has deliberately joined (`caucus_join_channel`) in this process. The tools take an optional `channel` routing arg (a per-call override, not a re-bind — the posting home, hence the hook's checkpoint, never moves); a write naming an un-joined room is rejected before the backbone is touched, with a value-free error (ADR-C12). The tool descriptions tell the model to post *sparingly* into another team's room. This is an addendum, not a reversal: the default is still your own room and silence; cross-room posting is the deliberate, auditable exception — a bare read of another room does NOT open the gate for posting into it. **Scope of the join-gate (important):** it is a **client-side guardrail that constrains the model**, not a server-enforced authorization boundary that constrains the *principal*. The backbone has no concept of "join": per ADR-C9, write authorization is *"any valid token → any existing channel"*, gated on token validity, not channel membership. The `joinedChannels` set only stops a well-behaved MCP session from spraying posts into rooms it never followed (the ADR-C6 noise / ADR-C12 leak-surface mitigation the CAU-77 decision asked for); a custom or malicious client holding a team token can already write to any room regardless. This is acceptable under the ADR-C9 intra-team single-server trust model (every token-holder is already trusted to read and write the shared log) — it must **not** be mistaken for a tenancy or access-control boundary.

## ADR-C7 — Multi-principal identity: agent → human, anchored server-side ✅
**Decision.** Every message is stamped `agent-id` + `human owner`. For MVP, identity comes from a per-session join token (shared team secret or simple issued token) mapped and anchored server-side so the owner can't be forged. A token-issuer service and cross-org identity are deferred (M2).
**Alternatives.** Flat nicks (scuttlebot's model — no human binding); trust client-asserted owner (spoofable).
**Why.** Multi-principal identity *is* the wedge — the room must know which teammate stands behind each agent. Anchoring server-side prevents spoofing without building an issuer prematurely.

**Addendum (CAU-20, 2026-06-20) — issuer activated; static map preserved as a seed. ✅**
**Decision.** Activate the token-issuer deferred in the base decision. The shared backbone server (ADR-C9 — still ONE loopback server, unchanged) gains an in-process **issuer**: an authenticated, loopback-only control surface to **mint** and **revoke** (and **rotate** = mint-then-revoke) per-agent bearer tokens at runtime, instead of requiring every credential in `CAUCUS_TOKENS` at boot. Server-side identity anchoring — the load-bearing property — is **preserved verbatim**: an issued token resolves to a server-held `{agent_id, owner}` the client never supplies and cannot forge; the write routes still overwrite any client-claimed identity. The mcp-server write-path firewall is untouched (the client still presents only an opaque bearer). The issuer and `CAUCUS_TOKENS` are **unified behind one `resolveToken` lookup**: `CAUCUS_TOKENS` becomes the **boot seed** (static, immutable, non-revocable entries), and the issuer holds **dynamic** entries layered over it in the same digest-keyed store; a revoked dynamic token resolves to nobody. **Token store is EPHEMERAL (ADR-C2 unchanged)** — process-memory only; a restart loses minted tokens, seeded tokens reload from env. **Fail-closed preserved** — empty store authorizes nobody; minting requires an admin credential (`CAUCUS_ADMIN_TOKEN`) and is reachable only over loopback; an unset admin token disables the control routes entirely. **Secret hygiene preserved (ADR-C12)** — a minted token is returned exactly once and never logged/echoed/re-readable; the store keeps only the SHA-256 digest. **Out of scope (still deferred):** cross-org/multi-server federation (ADR-C9), external token formats (JWT/OAuth), PKI, token TTL/auto-expiry.
**Note (CAU-128, 2026-06-23) — control-plane audit line is in-scope and secret-free.** Each mint/revoke/rotate (success or failure on an enabled surface) now emits exactly one structured **stderr** audit line (`{op, agent_id, owner, digest, ts, result}`, default ON via `CAUCUS_ADMIN_AUDIT`), closing the CAU-20 review's NOTE-2 (silent control-plane). It carries only the truncated SHA-256 **digest** of the affected token — consistent with the digest-only store — and **never** the plaintext minted token or the admin credential (ADR-C12); it goes to **stderr only**, never stdout, never the channel log (ADR-C6); a disabled control surface emits nothing (fail-closed no-op). No schema change, no new route.
**Alternatives.** (1) Keep static-only — rejected: can't rotate a leaked token or onboard mid-incident without downtime. (2) External issuer (OAuth/JWT) — rejected: a second trust boundary and key-management surface, breaks the ADR-C9 single-server model, premature for intra-team. (3) Persist tokens to disk — rejected: reopens ADR-C2 and reintroduces the on-disk-secret risk ADR-C14 avoids.
**Why.** Static `CAUCUS_TOKENS` means every credential change is a server restart and an env edit — untenable once a team rotates a leaked token or adds a teammate mid-investigation. A runtime issuer makes the identity wedge operationally real without expanding the trust boundary: loopback, ephemeral, fail-closed, anchored.

## ADR-C8 — Seatbelts: rate limit + loop/duplicate detection ✅
**Decision.** Per-agent posts/min cap (over-cap rejected with an actionable error the agent sees) and loop/dup detection (drop near-identical consecutive posts from the same agent).
**Why.** Autonomous posting plus agents reacting to each other is a predictable flood/loop risk that would wreck the calm-observation promise and burn tokens. Cheap to enforce in a backbone we own.

## ADR-C9 — Intra-team, single shared server; no federation in v1 🕓
**Decision.** One shared backbone, one team/org. No cross-org channels or multi-server federation in v1.
**Why.** The incident wedge is intra-team — everyone's on the same Slack already. Single-server keeps identity, claims, and the log trivially consistent and sidesteps the hardest problem (federation), which becomes far-future once there's pull.

## ADR-C10 — MVP = the two-terminal claim handoff ✅
**Decision.** v1 ships the smallest slice that proves the point: 2–3 Claude Code sessions in one ephemeral channel, claim-based dedup, hook-driven awareness, a human-injected steer propagating, and a seatbelt blocking a loop. (Full M1 demo definition in the [Roadmap](ROADMAP.md).)
**Why.** That single beat — an agent visibly avoiding redundant work because it saw another's claim — is the entire value, and it's the README demo and contributor magnet.

## ADR-C11 — Validate demand before building the backbone ✅
**Decision.** The backbone build (CAU-4 onward) is **gated on two cheap validation probes**: **Probe A** — interview 6–8 Claude-Code-using SRE/eng leads on whether concurrent per-engineer agent investigations happen today (CAU-22); **Probe B** — Wizard-of-Oz the claim/finding discipline in Slack with a human relay and *no backbone* (CAU-23). The hook-capability spike (CAU-24) and substrate spike (CAU-2) run in parallel as technical de-risking.
**Alternatives.** Build first, validate by dogfooding (faster, but spends backbone effort on a swarm pattern that may not exist at scale yet).
**Why.** The load-bearing assumption — that multiple engineers each run their own Claude Code on one investigation *today* — is admittedly emerging, not proven. A purpose-built backbone is premature spend if the concurrency isn't there. Probes are days, not weeks, and can kill or redirect cheaply. This also softens the earlier "design locked" overclaim: the *design* is set; *demand and the key technical pillars* are under validation.
**Amendment (2026-06-03) — probes waived by owner.** The owner decided not to run Probe A (CAU-22) or Probe B (CAU-23); both are closed **not planned**. No interview or Wizard-of-Oz data was collected, and **the load-bearing assumption remains unvalidated**. The gate on the backbone build (CAU-4+) is **lifted by owner decision, at owner-accepted risk**. Validation reverts to the alternative this ADR originally listed: **build first, validate by dogfooding** — the MVP itself (ADR-C10's two-terminal claim handoff, used on real investigations) becomes the demand probe. If dogfooding shows the multi-engineer-concurrent-agents pattern isn't real, revisit the wedge before any M2 spend. Nothing in this amendment claims demand was demonstrated.

**Dogfood verdict (2026-06-10) — GO, pending owner ratification.** The dogfood probe (CAU-85) replaced the waived CAU-22/23 as the demand gate. One real, non-scripted security investigation was run through the war room with two concurrent agent sessions; claim-before-work dedup, hook passive-awareness, and a session→session human steer all demonstrably fired (the steer redirected the second session on a bare "Continue." prompt, with no prompt-level mention). Evidence-based verdict: **GO** — see [the recorded verdict on CAU-85](https://github.com/basilmentorcruise/caucus/issues/85#issuecomment-4670225355) for the full AC checklist, transcript excerpts, and friction list. M2+ prioritization (CAU-16–20) stays frozen until the **owner ratifies or overrides** this verdict.

## ADR-C12 — Secret-leak hygiene is a first-class concern ✅
**Decision.** Treat the channel as a place secrets can leak: agents post diagnostic output (logs, tokens, customer data) into a **shared, persisted, append-only** log that propagates to everyone. v1 must ship a documented trust boundary, "what not to post" guidance, and a mitigation stance (CAU-26 / SECURITY.md). Borrow airc's AEAD associated-data binding of `{from,to,ts,channel}` — in our schema's field names: `{agent_id, owner, to, ts, channel}` (airc's `from` maps to the `agent_id`+`owner` pair) — so the backbone can't silently re-route a message.
**Why.** The incident wedge is exactly where sensitive output flies around. Silence on this is a real adoption blocker for security-conscious teams and a genuine leak vector. It was an unlisted non-goal/threat in the original docs.

## ADR-C13 — First-class `steer` (human-directive) message type ✅
**Decision.** Add `steer` as a distinct `type` enum value (NOT a `directive` flag on `note`) for a **human-injected directive** — one principal's human context crossing to *another* principal's agent (the wedge's signature moment, vision principle #1). It carries no new fields: a steer is an ordinary non-`claim` message and MAY carry `status: needs-response` (no new status values, no claim interaction, no verbosity change vs. ADR-C6). The MCP server adds a thin `caucus_steer` convenience tool (fixing `type: "steer"`, mirroring `caucus_post_finding`) so a human-relayed steer isn't free-typed, and the hook renders it on its own line with a leading, **descriptive** `▸ human directive:` marker. This **reverts the ADR-C5 note-as-steer idiom**: `note` is once again a freeform aside, and human directives use `steer`.

**Context-not-command guarantee.** A steer is rendered as *context to attend to*, never as an imperative the agent must execute. The marker is a fixed, server-emitted literal; the steer body is control-stripped and one-lined like every other body and rendered only as inert content *after* the marker — it cannot forge the marker, the `[caucus] ` line prefix, or the `=== … ===` delta frame (the frame is emitted only by the hook's `renderDelta`). A vision-guard test pins this against a prompt-injection body.

**Version-bump / compatibility stance — HARD CUTOVER.** This is a FROZEN-schema change, so `SCHEMA_VERSION` bumps **v0 → v1** (the schema was ratified and frozen in M0, CAU-3). The codec's version gate stays **exact-match**: decoding a `v: 0` message now throws `UnsupportedVersionError` (received 0, supported 1). This is safe today because **no production path re-decodes stored messages** — the backbone stamps `v` on write and replays stored object references on read without re-validation, and the MVP store is in-memory (no durable, re-decoded log). The blast radius is therefore limited to hardcoded `v: 0` *test fixtures*. **A future durable backbone that persists wire bytes across a version boundary will need a read-time migration** (or a widened, multi-version gate) before it can replay a v0 log into a v1 build — that is explicitly out of scope here and must be designed when durability lands.

**Identity.** No new identity field. ADR-C7 anchoring already answers "whose human steered": a steer is stamped with the relaying session's server-anchored `agent_id`/`owner` (the human can't be forged), so the room knows which teammate's context crossed.

**Alternatives.** (1) A `directive: true` flag on `note` — rejected: it hides the wedge's signature event behind a boolean, makes rendering and the tool surface conditional, and still needs a version bump to add the field. (2) Keep approximating steers as `note` + `needs-response` (the CAU-85 dogfood reality, friction #4) — rejected: agents had to infer intent, so the one-human-context-crossing-to-another-agent moment was illegible. (3) A soft/multi-version gate that also accepts v0 — deferred: unnecessary while nothing re-decodes stored messages, and it would weaken the frozen-schema contract; revisit when a durable backbone needs read-time migration.

**Why.** Making the human steer a first-class, distinctly-rendered, never-auto-executed type makes the wedge legible and unmistakably *context, not command*. Doing the version bump now — early in M2, while consumers are few — is far cheaper than later. Secret hygiene is unchanged (ADR-C12): a steer is an ordinary persisted message and the tool copy forbids secrets.

## ADR-C14 — Shared ephemeral evidence store: the `artifact` URI may point at a backbone-hosted payload ✅
**Decision.** Add a content-addressed, **ephemeral, intra-channel** payload store to the backbone that the existing `artifact` field's URI may point *into* — a deliberate, bounded reversal of the prior pointer-only stance ("`artifact` is a pointer, not a payload"). Storage is a `sha256 → bytes` map owned **per-channel** by the backbone (a blob map on `ChannelState`), mirroring the ADR-C2 in-memory log and tracking the channel lifecycle exactly: a blob is keyed by the SHA-256 of its own bytes, stored once (dedup), and verified on write (`sha256(body)` MUST equal the addressed digest). **Lifecycle = channel = process exit.** There is NO durability, NO archival, NO on-disk persistence, NO explicit delete, NO versioning, and NO GC/TTL machinery — "ephemeral" means *process lifetime*: blobs die when the backbone process exits, alongside the channels and the log. Transport is two routes on the single shared server (ADR-C9): a `PUT` (token-gated like `append`, fail-closed — an empty/unknown token authorizes nobody) that buffers raw bytes, verifies the digest, and is idempotent (201 on first store, 200 if the blob already exists); and a `GET` (tokenless within the trust boundary, like `readSince`) that serves the raw blob as opaque `application/octet-stream`, 404 when absent. The URI carried in `artifact` is a **logical**, host-agnostic `caucus://artifact/<channel>/<sha256>`; the MCP client resolves it to `${CAUCUS_URL}/channels/<channel>/artifacts/<sha256>` at fetch time **against its own already-validated `CAUCUS_URL`** — it never dials a caller-supplied host, so there is no new SSRF surface (this reconciles the CAU-75 URL-validation posture: the fetcher trusts only its own wiring). It rides the existing `artifact` string with NO schema version bump: the validator already accepts `caucus://…` (non-empty, no control characters, ≤ `MAX_FIELD_CHARS`). Cooperative byte bounds, per CAU-74/83, are enforced as exported constants: per-blob ≤ 1 MiB, per-channel ≤ 16 MiB, global ≤ 128 MiB, with an over-cap upload rejected `413` **mid-stream** (no GB buffering) and running per-channel + global byte totals maintained as blobs are stored. There is no read-amplification: a `GET` serves exactly one blob, and `readSince` carries only the bounded URI, never the bytes.

**ADR-C12 stance — boundary UNCHANGED, surface acknowledged.** A blob is the same shared-log leak surface as a message `body`: it lives behind the same "never post secrets" boundary, under the same size caps, and is equally ephemeral. The upload tool copy forbids secrets exactly as the post tools do. The blob is treated as **opaque, binary-safe bytes** — it is NEVER rendered into a hook line (the hook already suppresses `artifact` to a bare `↗artifact` marker, and that is left unchanged), so the control-character / no-echo discipline is preserved by construction: the bytes are never echoed into any model context or TTY. This **refines, not reverses, ADR-C12**: the channel still is not a safe place for secrets, including uploaded ones. The agent surface is two tools — `caucus_upload_artifact` and `caucus_fetch_artifact` — scoped to sharing a repro/evidence across sessions and machines, nothing more.

**Alternatives.** (1) Stay pointer-only — rejected: the CAU-85 dogfood verdict named "evidence stays in `/tmp`, unshareable" as friction #3, the **biggest** gap; the room shares conclusions but not reproducible evidence. (2) An external object store (S3 / similar) — rejected: durability is explicitly out of scope, it introduces a second trust boundary, and it breaks the ADR-C9 single-shared-server model. (3) The filesystem / a shared tmp dir — rejected: a path is unshareable across machines and an on-disk blob reintroduces the on-disk-secret risk the in-memory model avoids. (4) Inline the bytes in the message `body` or `artifact` field — rejected: it blows the size caps, forces base64 into a control-char-rejecting field, and would push payload bytes through the hook render. (5) An arbitrary HTTP-URL `artifact` — rejected: it is machine-specific and reopens the CAU-75 SSRF surface (the fetcher would dial a caller-supplied host).

**Why.** This is the one architectural expansion M2 ratified (owner-ratified, 2026-06-10, "build it, ephemeral + ADR-first"), and dogfood evidence justifies it. Bounding it to content-addressed + ephemeral + process-lifetime + cooperative caps serves the real need — cross-session, cross-machine evidence-sharing so a finding's repro travels with it — without taking on durability. The leak boundary does not move: ADR-C12 is refined, not reversed.

## ADR-C15 — Human observability via a read-only SSE log-tail ✅
**Decision.** Caucus ships one read-only streaming endpoint, `GET /channels/:channel/stream`, on the existing shared server, emitting each new log message as a sanitized `text/event-stream` JSON frame, driven by a server-internal `readSince` poll loop (the backbone has no server-push primitive; it polls at human cadence, ~1s). It is a **human read-tail, NOT the deferred real-time SDK (ADR-C4/CAU-16)** and not an agent transport — agents remain turn-based via the hook; the stream is one-directional, write-free, and never touches the turn loop. It is **tokenless-read within the ADR-C9 boundary, loopback-default** (inherits the existing bind posture + `bindExposureWarning`); the trust model is unchanged because it is `readSince` with the socket held open. Stream start defaults to the `subscribe`-minted head (future-only, calm-feed); an optional `?since=<cursor>` (the same integer `readSince` takes) is in MVP for mid-incident catch-up. The route **consciously exempts itself from the CAU-75 slowloris request/keepalive timeouts** and instead bounds resources with periodic heartbeat comments and a **max-concurrent-stream cap (32, exported/tunable; N+1th → 503)** — the exemption is scoped to this route only and must not weaken the JSON routes' timeouts.

**ADR-C12 stance — satisfied by reuse, not extension.** The frame payload is the same field-sanitized JSON `caucus_read_channel` already returns (`stripControlChars*`), produced via the same path (asserted byte-identical), so there is no new serializer and no new leak class. `text/event-stream` is not a DOM — no HTML/XSS surface (that arrives only with the deferred web viewer, CAU-118, which gets its own ADR-C12 review). No schema change, no version bump.

**Alternatives.** (A) A read-only HTML web viewer — deferred to CAU-118 (first browser/XSS surface; a real ADR-C12 expansion; structurally built on this stream). (C) An Ergo/IRC mirror — rejected: reopens the closed ADR-C2 "not an Ergo fork" decision against no demonstrated IRC demand, adds a second auth-less network listener (ADR-C9), heaviest maintenance/leak burden; stays the documented reversible fallback, unbuilt.

**Why.** The incident-commander persona (the CAU-19 retention story) benefits from a live, no-agent-required window on the war room. SSE is the minimal surface that proves it — rides seams that already exist and are already authorized, adds no new leak class, keeps ADR-C4 intact, and needs no new client (`curl -N | jq` is the demo).

## Open risks tracked against these decisions
- **Substrate spike (ADR-C2):** ✅ resolved — **GO**. Atomic first-write-wins claim, cursor survival across discrete MCP calls, and acceptable turn-based latency all empirically confirmed; transport = HTTP + cursor polling, store = append-only event log + projections. See [docs/spikes/cau-2-substrate/verdict.md](spikes/cau-2-substrate/verdict.md) (CAU-2).

- **Context flooding (ADR-C3/C4):** cap injected delta; overflow to an explicit read.
- **Load-bearing assumption (ADR-C1):** ⚠️ consciously unvalidated — demand probes waived by owner (ADR-C11 amendment, 2026-06-03); validation deferred to dogfooding the MVP.
